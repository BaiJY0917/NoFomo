# RSS 每日摘要推送工具设计

- 日期：2026-04-11
- 项目：NoFomo
- 状态：设计已确认，待审阅文档

## 1. 背景与目标

这是一个新建目录中的新工具，目标是帮助用户每天处理来自多个自媒体/社区平台的公开内容更新。

第一版聚焦于以下场景：

- 监控外部公开账号/频道，而不是登录自有账号后台
- 数据来源限定为可稳定获取的 RSS / 公开订阅源
- 平台范围包括：Weibo、V2EX、X、Telegram Channel
- 主要输出是“摘要整理”，不是简单罗列原始帖子
- 当内容命中用户感兴趣的关键词时，需要在日报中重点标出，并保留原始链接，方便跳转回原平台阅读
- 日报的交付方式是 Telegram 推送
- 用户后续希望基于 Telegram 中的阅读反馈逐步优化兴趣判断，但第一版仅记录反馈，不自动学习

## 2. 第一版范围

### 2.1 In scope

第一版实现以下能力：

1. 配置一组 RSS 来源
2. 每天定时拉取所有启用来源的新内容
3. 对新内容进行去重和标准化
4. 生成适合阅读的摘要
5. 按关键词识别重点内容
6. 生成日报并通过 Telegram Bot 推送
7. 在重点内容和普通内容中保留原始链接
8. 记录用户在 Telegram 中对内容的简单反馈
9. 将抓取结果、日报、反馈保存到本地文件中

### 2.2 Out of scope

第一版明确不做以下内容：

- 非 RSS 的网页抓取或 API 兜底采集
- 登录任何平台后台
- 采集私有或受限数据
- 自动学习、自动调整关键词或推荐模型
- 多渠道推送（邮件、网页、数据库服务等）
- 长期常驻服务

## 3. 产品形态

第一版产品形态是一个以“单次执行 CLI”为核心的本地工具，包含两个入口：

1. `run-digest`
2. `sync-feedback`

其中：

- `run-digest` 由外部调度在设定时间自动执行
- `sync-feedback` 由用户在阅读完日报后按需手动执行
- 两个入口都在完成各自任务后退出

`run-digest` 的执行流程为：

1. 读取来源配置
2. 拉取 RSS 更新
3. 处理新内容
4. 生成日报
5. 推送到 Telegram
6. 保存本地归档与日志
7. 退出

`sync-feedback` 的执行流程为：

1. 拉取 Telegram Bot 尚未消费的消息更新
2. 解析 `/like <item_id>` 与 `/dislike <item_id>`
3. 校验 item_id
4. 保存反馈记录与 offset 状态
5. 退出

该形态的重点是简单、稳定、便于排查。未来如果需要更高频运行或更强交互能力，可以在不推翻核心结构的情况下扩展执行方式。

## 4. 核心设计

### 4.1 总体运行形态

第一版采用两个手动可理解、职责分离的入口：

1. `run-digest`
   - 由系统 cron / Windows 任务计划程序每天触发一次
   - 单次执行后退出
   - 负责抓取 RSS、去重、标准化、摘要、关键词识别、生成日报、推送 Telegram、保存本地归档与日志

2. `sync-feedback`
   - 由用户在阅读完日报后按需手动触发
   - 单次拉取并处理 Telegram Bot 尚未消费的消息更新后退出
   - 负责解析 `/like <item_id>` 与 `/dislike <item_id>` 命令，并记录反馈

第一版不引入常驻进程，不使用 webhook，不要求 Bot 长时间轮询。这样可以优先验证“日报摘要是否有价值”，并保持本地工具的简单性与可排障性。

### 4.2 来源配置模块

负责维护所有订阅源配置。每个来源至少包含：

- `id`
- `platform`：weibo / v2ex / x / telegram
- `name`：来源显示名
- `rss_url`
- `enabled`

职责边界：

- 只负责描述“监控哪些来源”
- 不负责抓取、摘要、推送逻辑

### 4.3 抓取与去重模块

负责：

- 定时拉取所有启用 RSS
- 解析新增条目
- 对条目进行去重
- 只输出本次执行中真正的新内容

第一版去重依据优先级如下：

1. 原始链接
2. 来源 ID + 条目唯一标识（如 GUID）
3. 标题 + 发布时间的组合作为兜底

去重记录保存在本地，用于避免重复推送历史内容。

需要注意的执行策略：

- 只有条目成功进入本次待发送日报后，才允许在最终发送成功后写入去重状态
- 如果运行过程在发送前失败，不应提前把条目标记为已处理，避免后续漏推

### 4.4 内容标准化与摘要模块

由于不同 RSS 字段差异较大，需要先把条目统一成内部数据结构。统一后的字段至少包括：

- `source_id`
- `platform`
- `source_name`
- `title`
- `published_at`
- `url`
- `raw_summary`
- `normalized_text`
- `summary_short`
- `summary_long`

在统一结构上生成日报用摘要。摘要策略如下：

1. 优先复用 RSS 已提供的高质量摘要
2. 如果摘要缺失或质量较差，则从正文片段中提取简要摘要
3. 第一版默认生成简洁摘要，目标是提高可读性，而不是替代原文
4. `summary_long` 用于“重点关注”部分，`summary_short` 用于“普通更新”部分

### 4.5 兴趣识别模块

第一版采用基于关键词的规则判断。关键词由用户配置，命中范围包括：

- 标题
- 摘要
- 规范化后的正文片段

命中后将条目标记为“重点关注”，并记录命中的关键词列表。

第一版不做：

- 权重学习
- 自动加减词
- 相似内容聚类推荐
- 基于历史偏好的自动排序模型

### 4.6 报告生成与 Telegram 推送模块

日报固定分成三部分：

#### 今日概览

包含：

- 今日抓取到的新内容总数
- 涉及的来源数量
- 重点关注条目数量
- 抓取异常来源数量（如果有）

#### 重点关注

仅展示命中关键词的内容。每条至少包含：

- 条目 ID（用于反馈）
- 来源平台 + 来源名
- 标题或主题句
- 1 到 3 句摘要
- 命中的关键词
- 原始链接
- 反馈命令提示：`/like <item_id>`、`/dislike <item_id>`

#### 普通更新

展示其余未命中关键词的内容。每条至少包含：

- 条目 ID（用于反馈）
- 来源名
- 标题
- 一句话摘要
- 原始链接
- 反馈命令提示：`/like <item_id>`、`/dislike <item_id>`

Telegram 推送的目标是让用户先在消息里浏览摘要，再按需点击原始链接进入平台阅读原文。为避免反馈关联歧义，日报中的每条内容都需要带稳定的条目 ID，供用户在 Telegram 中引用。

第一版消息组织策略如下：

- 先发送一条“今日概览”消息
- 重点关注内容按单条消息发送，避免长消息截断
- 普通更新内容也按单条消息发送，保持内容与反馈 ID 一一对应
- 第一版不使用 inline keyboard，不修改历史消息状态，只在消息正文中提供命令式反馈提示

### 4.7 反馈记录模块

用户希望后续通过 Telegram 中的反馈逐渐优化兴趣判断，因此第一版先实现“记录反馈”。

支持的反馈形式固定为命令式输入：

- `/like <item_id>`
- `/dislike <item_id>`

第一版记录的信息包括：

- 对应条目 ID
- 所属日报日期
- 反馈类型（感兴趣 / 不感兴趣）
- 反馈时间
- 条目来源
- 当时命中的关键词
- Telegram 用户 ID / Chat ID / 消息 ID

第一版仅记录反馈，不自动修改关键词或推荐逻辑。

反馈同步策略如下：

- `sync-feedback` 启动时从上次保存的 Telegram offset 继续拉取更新
- 只处理结构合法且 item_id 可校验通过的反馈命令
- 只有在某条 update 成功处理后才推进 offset，避免异常时误跳过未处理消息
- 反馈记录采用追加写入方式保存，便于后续分析与审计

## 5. 数据流

完整数据流分成两条：

### 5.1 日报生成链路（`run-digest`）

1. 读取来源配置
2. 拉取所有启用 RSS
3. 解析与去重
4. 标准化内容结构
5. 生成摘要
6. 按关键词识别重点条目
7. 生成日报正文
8. 推送 Telegram 消息
9. 保存日报归档、去重记录与运行日志

### 5.2 反馈同步链路（`sync-feedback`）

1. 读取 Telegram offset 状态
2. 拉取尚未消费的 Bot 更新
3. 解析 `/like <item_id>` 与 `/dislike <item_id>` 命令
4. 校验 item_id 是否能在历史日报中找到
5. 写入反馈记录
6. 推进并保存 offset
7. 记录同步日志

## 6. 存储设计

第一版使用本地文件存储，不引入数据库。

建议保存以下几类数据：

### 6.1 来源配置

例如：`config/sources.yaml`

保存监控源定义，包括平台、名称、RSS 地址、启用状态。

### 6.2 关键词配置

例如：`config/keywords.yaml`

保存用户的兴趣关键词列表。

### 6.3 Telegram 配置

例如：`config/telegram.yaml`

保存 Bot Token、Chat ID 以及运行所需的 Telegram 相关配置。

### 6.4 去重状态

例如：`data/seen_items.json`

保存已处理内容的唯一标识，用于避免重复摘要和重复推送。

### 6.5 Telegram 拉取状态

例如：`data/telegram_state.json`

保存上次成功处理的 update offset，用于 `sync-feedback` 增量同步，避免重复消费消息。

### 6.6 日报归档

例如：`data/reports/YYYY-MM-DD.json`

保存当天生成的结构化日报内容，用于回看、校验反馈 item_id 与后续手动重发。

### 6.7 反馈记录

例如：`data/feedback.jsonl`

保存用户对具体条目的反馈记录。采用 JSON Lines 追加写入，以降低修改成本并便于后续分析。

### 6.8 运行日志

例如：`data/logs/YYYY-MM-DD.log`

保存抓取失败、解析失败、发送失败、反馈同步失败等运行信息。

## 7. 配置项

第一版至少需要三类配置：

### 7.1 RSS 来源配置

每个来源需要：

- 来源名称
- 平台类型
- RSS 地址
- 是否启用

### 7.2 兴趣关键词配置

维护一组用于识别重点内容的关键词，例如 AI、agent、自动化、出海、增长等。

### 7.3 Telegram 推送配置

需要：

- Bot Token
- Chat ID
- 推送时间
- feedback 同步所需的 update offset 状态文件路径（默认可固定在 `data/telegram_state.json`）

## 8. 错误处理策略

第一版只处理高价值的常见错误，避免复杂化。

### 8.1 RSS 拉取失败

处理方式：

- 记录失败日志
- 不阻塞其他来源继续执行
- 在日报底部的异常区列出失败来源

### 8.2 条目格式异常

处理方式：

- 跳过异常条目
- 记录错误详情
- 不让单条坏数据中断整个任务

### 8.3 Telegram 发送失败

处理方式：

- 保留已生成日报到本地归档
- 记录发送失败日志
- 不写入本轮新增条目的去重状态
- 支持后续手动重发当天日报

### 8.4 feedback 同步失败

处理方式：

- 记录同步失败日志
- 保留旧 offset，不前移
- 避免未成功处理的 Telegram 更新被错误跳过

### 8.5 无关键词命中

处理方式：

- 正常发送日报
- “重点关注”部分可为空
- “普通更新”部分照常输出

## 9. 成功标准

第一版达成以下目标，即视为可用：

1. 用户可以维护一组 RSS 来源
2. 工具可以每天稳定拉取新内容
3. 相同内容不会重复推送
4. 日报能区分重点关注和普通更新
5. 所有日报条目附带稳定 item_id 与原始链接
6. Telegram 推送稳定可用
7. 用户可通过 `/like <item_id>` 与 `/dislike <item_id>` 提交反馈
8. 反馈可被记录并归档

## 10. 未来扩展方向

以下方向明确留到后续版本，不在第一版实现：

- 增加非 RSS 数据源适配器
- 基于反馈半自动优化关键词
- 增加来源优先级和主题分组
- 支持多频道推送
- 从本地文件迁移到 SQLite 等更强存储

## 11. 当前项目上下文

当前仓库目录为新建目录，尚无现有代码和既有约束。第一版设计因此优先追求：

- 单一职责清晰
- 易于从零实现
- 易于验证和手动排障
- 不为未来假设需求过度设计

## 12. 结论

第一版将实现一个基于 RSS 的每日内容监控与摘要推送工具：

- 输入：Weibo、V2EX、X、Telegram Channel 的 RSS / 公开订阅源
- 处理：定时抓取、去重、标准化、摘要、关键词筛选
- 输出：Telegram 每日报告
- 交互：保留原始链接，支持简单反馈记录

该设计刻意保持边界清晰，优先确保“每天稳定跑通并对用户有用”，而不是一次性覆盖所有可能扩展方向。
